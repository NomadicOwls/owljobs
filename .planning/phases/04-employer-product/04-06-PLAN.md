<objective>
Ship migration 0008 — Row-Level Security for the employer dashboard. This is the second of two security-critical migrations in Phase 4 (0007 ships schema + Auth Hook in plan 01; 0008 ships RLS policies that use the verified JWT path).

**Phase Goal (user story):** **As an** ingested employer, **I want to** claim my auto-generated company page, log in via magic link, feature my jobs, and see 30-day analytics, **so that** I can evaluate the product before being billed in Phase 5.

Why this is its own plan (not folded into 01): the RLS expressions depend on the exact JWT path for `employer_id`, which is version-dependent and verified only at runtime against a live Supabase project. Plan 02 Task 3 produces `jwt-path-verification.md` recording the verified expression. Plan 06 reads that file and substitutes the `<<JWT_EMPLOYER_ID>>` placeholder before the SQL is applied.

Mitigates T-04-12 (JWT claim path mismatch → silent RLS bypass — RESEARCH Critical Finding 4 / Pitfall 2). If the wrong path is used, all RLS policies return zero rows and the dashboard silently shows an empty job list. The verification gate prevents that.

Purpose: Enforce employer-scoped access at the database layer (defense-in-depth with the app-level IDOR check in plan 04).
Output: 0008_employer_rls.sql + operator confirmation that it has been applied.
</objective>

From `.planning/phases/04-employer-product/jwt-path-verification.md` (written by plan 02 Task 3):
- `JWT_PATH_EXPRESSION`: a backtick-quoted SQL expression — either `auth.jwt() ->> 'employer_id'` (flattened) OR `auth.jwt() -> 'app_metadata' ->> 'employer_id'` (nested). The migration uses exactly the value recorded there.

From migration 0001 (schema reference):
- `wind_turbine.employers.id TEXT PRIMARY KEY` (SHA-256 hash; 64-hex char)
- `wind_turbine.jobs.employer_id TEXT` (FK to employers.id)
- All niche-scoped tables live under the substituted `«wind_turbine»` schema name.

From migration 0007 (plan 01):
- `public.employer_users (id UUID PK, auth_id UUID NULLABLE, employer_id TEXT NOT NULL, niche_id TEXT NOT NULL, ...)` — public schema, no per-niche substitution.
- Existing public reads on `wind_turbine.employers` and `wind_turbine.jobs` are performed by the `supabasePublic(env)` anon-key client. Once RLS is enabled, the anon role needs an explicit "public read of active jobs / employers" policy or those pages 500.

Critical: enabling RLS on a table with NO policies blocks ALL access. We MUST ship the public-read policies alongside the employer-scoped policies, or /jobs and /employers/[slug] break.

The four-policy set for `wind_turbine.jobs`:
1. `public_read_active` — anon + authenticated: SELECT WHERE status='active' (mirrors the current public visibility)
2. `employer_read_own` — authenticated: SELECT WHERE employer_id = JWT employer_id (lets employer see expired/draft jobs in dashboard)
3. `employer_update_own` — authenticated: UPDATE WHERE employer_id = JWT employer_id (featured_until toggle)
4. service_role bypasses RLS by default — no policy needed for supabaseAdmin writes (ingest, claim, featured toggle)

The two-policy set for `wind_turbine.employers`:
1. `public_read` — anon + authenticated: SELECT (employer profile pages are public — no filtering)
2. `employer_self_update` — DEFERRED to Phase 5 when profile editing unlocks; no policy in 0008

The one-policy set for `public.employer_users`:
1. `self_read` — authenticated: SELECT WHERE auth_id = auth.uid() (dashboard "who am I" lookups; rarely needed since employer_id is in the JWT, but useful for membership debugging)
</interfaces>

```
-- OwlJobs Phase 4 — Employer RLS Policies
-- Paste into Supabase SQL Editor, OR apply via `supabase db push`.
-- Replace every occurrence of «wind_turbine» with your niche's schema name before running.
-- The provision script (pnpm niche:provision <id>) generates a pre-substituted copy.

-- This migration ships RLS policies that use the JWT path verified in
-- .planning/phases/04-employer-product/jwt-path-verification.md (plan 02 Task 3).
-- DO NOT apply 0008 before 0007 is applied AND the Auth Hook is enabled.

-- ─── public.employer_users ────────────────────────────────────────────────────
ALTER TABLE public.employer_users ENABLE ROW LEVEL SECURITY;

DROP POLICY IF EXISTS employer_users_self_read ON public.employer_users;
CREATE POLICY employer_users_self_read
  ON public.employer_users
  FOR SELECT
  TO authenticated
  USING (auth_id = auth.uid());

-- ─── wind_turbine.employers (public profile pages) ────────────────────────────
-- NOTE: RLS already enabled in migration 0002; `public_employers` policy covers anon.
-- This migration only adds an `authenticated` read for logged-in employers (mirrors anon view).
DROP POLICY IF EXISTS employers_auth_read ON wind_turbine.employers;
CREATE POLICY employers_auth_read
  ON wind_turbine.employers
  FOR SELECT
  TO authenticated
  USING (TRUE);

-- NOTE: employer profile UPDATE policy is intentionally NOT created here.
-- Profile editing is locked in Phase 4 (D-06) and writes go through supabaseAdmin
-- (service_role bypasses RLS). Phase 5 will add an UPDATE policy gated on
-- subscription_active = true AND tier IN ('starter','growth','partner').

-- ─── wind_turbine.jobs (employer dashboard reads) ─────────────────────────────
-- NOTE: RLS already enabled in migration 0002; `public_relevant_jobs` policy covers anon
-- with the (classification_score >= 0.6 OR sponsored) filter. We do NOT replace it.
-- This migration adds:
--   1. authenticated read mirroring the anon policy (so logged-in employers browsing /jobs
--      see the same listings anon visitors do)
--   2. employer read of their own jobs (including drafts/expired) for the dashboard
--   3. employer UPDATE of their own jobs (featured_until toggle)

-- 1. Authenticated read of the same set anon sees (defence-in-depth — RLS would otherwise
--    block logged-in users from /jobs because policies are role-scoped)
DROP POLICY IF EXISTS jobs_auth_read_public ON wind_turbine.jobs;
CREATE POLICY jobs_auth_read_public
  ON wind_turbine.jobs
  FOR SELECT
  TO authenticated
  USING (
    classification_score >= 0.6
    OR (is_sponsored AND (featured_until IS NULL OR featured_until > now()))
  );

-- 2. Employer reads their own jobs (including expired/inactive — dashboard view)
DROP POLICY IF EXISTS jobs_employer_read_own ON wind_turbine.jobs;
CREATE POLICY jobs_employer_read_own
  ON wind_turbine.jobs
  FOR SELECT
  TO authenticated
  USING (employer_id = (<<JWT_EMPLOYER_ID>>));

-- 3. Employer toggles featured_until on their own jobs (FEAT-03)
--    The app layer (plan 04 /api/jobs/[id]/featured.ts) additionally checks
--    employer_id match before UPDATE — defense-in-depth. RLS catches direct
--    PostgREST attempts that bypass the API.
DROP POLICY IF EXISTS jobs_employer_update_own ON wind_turbine.jobs;
CREATE POLICY jobs_employer_update_own
  ON wind_turbine.jobs
  FOR UPDATE
  TO authenticated
  USING (employer_id = (<<JWT_EMPLOYER_ID>>))
  WITH CHECK (employer_id = (<<JWT_EMPLOYER_ID>>));

-- service_role bypasses RLS by default; no policy is needed for the ingest worker,
-- the claim API, or the featured toggle API — they all use supabaseAdmin() and
-- write with the service key.

-- ─── Deferred (Phase 5) — DO NOT UNCOMMENT IN 0008 ────────────────────────────
-- -- Employer profile update — gated on subscription_active
-- CREATE POLICY employers_self_update_paid ...
-- -- Employer jobs INSERT (manual job posting) — Phase 5+
-- CREATE POLICY jobs_employer_insert ...
```
