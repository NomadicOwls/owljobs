    **Step A — frontmatter additions** (in the `---` script block at top, after the `employerInitials` constant):

    ```typescript
    // SEO-01 (D-15..D-18 + Pitfalls 4, 9): Conditional JSON-LD JobPosting structured data.
    // Render only when:
    //   - job has a non-null description (D-15 — thin content guard)
    //   - source is NOT adzuna or jsearch (Pitfall 4 — teaser/aggregator content)
    // jobLocation is conditionally spread when job.location truthy (D-16 — no inference)
    // validThrough uses Math.max(posted+30d, now+7d) (Pitfall 9 — future-date guard)
    const aggregatorSources = new Set<string>(["adzuna", "jsearch"]);
    const isAggregator = (job.job_sources ?? []).some((s) => aggregatorSources.has(s.source));
    const shouldEmitJsonLd = !isExpired && !!job.description && !isAggregator;

    const THIRTY_DAYS_MS = 30 * 24 * 60 * 60 * 1000;
    const SEVEN_DAYS_MS = 7 * 24 * 60 * 60 * 1000;
    const postedAtMs = job.posted_at ? new Date(job.posted_at).getTime() : Date.now();
    const validThrough = new Date(
      Math.max(postedAtMs + THIRTY_DAYS_MS, Date.now() + SEVEN_DAYS_MS),
    ).toISOString();

    const datePosted = job.posted_at
      ? new Date(job.posted_at).toISOString().split("T")[0]
      : new Date().toISOString().split("T")[0];

    const jsonLd = shouldEmitJsonLd
      ? {
          "@context": "https://schema.org",
          "@type": "JobPosting",
          "title": job.title,
          "description": job.description,
          "datePosted": datePosted,
          "validThrough": validThrough,
          "hiringOrganization": {
            "@type": "Organization",
            "name": employerName,
          },
          "url": canonicalUrl,
          ...(job.location
            ? {
                "jobLocation": {
                  "@type": "Place",
                  "address": {
                    "@type": "PostalAddress",
                    "addressLocality": job.location,
                    ...(job.country ? { "addressCountry": job.country } : {}),
                  },
                },
              }
            : {}),
        }
      : null;
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase DB → Astro SSR | Job description is pre-sanitized at ingest (sanitizeJobDescription); rendered via `set:html` |
| Public → Astro page response | JSON-LD body is JSON.stringify of structured fields (no user input concatenation) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | JSON-LD body injection | mitigate | JSON.stringify does NOT escape `<` per ECMA-262 — `</script>` in a job description is a known XSS vector for inline JSON-LD. Mitigation: `.replace(/</g, "\u003c")` after stringify. Browsers JSON.parse the unicode escape back to `<` correctly. |
| T-02-05 | Information Disclosure | Description content in JSON-LD | accept | Description is already public on the job page; including it in structured data adds no new exposure |
| T-02-06 | Repudiation | False structured data triggering Google manual action | mitigate | D-15 (no thin descriptions), Pitfall 4 (no aggregator content), D-16 (no location inference) all enforced via guards |

</threat_model>
