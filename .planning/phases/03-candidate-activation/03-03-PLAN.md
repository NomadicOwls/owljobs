<objective>
Vertical slice that fixes the unsubscribe action to soft-delete (preserve FK integrity from email_sends, enable clean re-subscribe). This closes the loop on CAND-02: the digest email's `List-Unsubscribe-Post` header (set by Plan 02) now points to a handler whose action behaves correctly.

Purpose: Without this plan, every one-click unsubscribe hard-deletes the subscriber row, orphaning historical `email_sends` records (FK constraint failure) and forcing users to re-enter their email + locations on re-subscribe.

Resolves three conflicts flagged in RESEARCH.md:
- **Conflict 2:** D-19 says "new endpoint /api/unsubscribe-oneclick" but `apps/web/src/pages/api/unsubscribe.ts` already exports a `POST` handler. Modify in place — do NOT create a new file. (RESEARCH §Conflict 2.)
- **Conflict 3:** D-19 says "token in POST body" but RFC 8058 §3.1 mandates token in URL, body is literal `List-Unsubscribe=One-Click`. Existing code reads `url.searchParams.get("token")` — already correct, do NOT change.
- **Conflict 4:** GET hard-deletes, D-20 soft-deletes POST — asymmetry. **Decision: migrate GET to soft-delete as well** (RESEARCH recommendation — FK integrity + re-subscribe UX, no downside).

Output:
- `apps/web/src/pages/api/unsubscribe.ts` — GET + POST both soft-delete
- `apps/web/test/unsubscribe.test.ts` — source-contract test asserting the change
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-01 | Information Disclosure | POST `/api/unsubscribe` response side-channel reveals whether a token corresponds to a real subscriber | mitigate | POST returns `new Response("OK", { status: 200 })` unconditionally — no branch on row-not-found. Re-subscribe flow does not require re-confirmation under D-20 since the row is preserved with `confirmed_at = NULL` — operator can re-trigger confirmation if needed; out of scope for Phase 3. |
